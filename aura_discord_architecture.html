Hub router — Aura owns all URLs
The URL never leaves /aura/hubs/:id. Routing, history, and navigation are entirely Aura-controlled. Discord never appears in the address bar. Hub metadata, member lists, and challenge state all load from Aura's own API — Discord is invisible to the user.
AuraChat UI — optimistic message flow
Emit message on send, display immediately with a "pending" visual state, then replace with the confirmed Discord message ID when the relay acknowledges it. Failures show a retry prompt. User perceives zero latency.
// 1. Optimistic insert const tempId = crypto.randomUUID(); addMessage({ id: tempId, content, status: 'pending' }); // 2. Relay to backend socket.emit('hub:message', { hubId, content, tempId }); // 3. Confirm via bot event socket.on('hub:message:ack', ({ tempId, discordId }) => { confirmMessage(tempId, discordId); });
AuraVoice — LiveKit replaces discord://
Embed the @livekit/components-react voice panel directly in Aura. LiveKit is an open-source WebRTC SFU — self-hostable, no Discord app required. Your backend provisions a LiveKit room per hub voice channel and issues short-lived JWT tokens. Voice is 100% in the browser tab.
import { LiveKitRoom, AudioConference } from '@livekit/components-react'; const { token } = await fetch(`/api/hubs/${hubId}/voice-token`).then(r=>r.json()); <LiveKitRoom serverUrl={LIVEKIT_URL} token={token}> <AudioConference /> </LiveKitRoom>
AuraPresence — Discord voice states mirrored in-UI
Your Discord bot subscribes to VOICE_STATE_UPDATE gateway events. When a member joins/leaves/mutes, the bot emits to your Aura socket, which updates a HubVoiceState map in Redis. Aura renders speaking indicators and the voice member list natively — users see rich presence without ever seeing Discord.
// Bot gateway handler client.on('voiceStateUpdate', (oldState, newState) => { const hubId = mapping.get(newState.channelId); if (!hubId) return; io.to(`hub:${hubId}`).emit('presence:update', { userId: newState.member.user.id, action: newState.channelId ? 'joined' : 'left', muted: newState.mute, deafened: newState.deaf, }); });
ChallengeEngine — bot as orchestrator
When a challenge starts in Aura, emit a socket event to backend, which instructs the Discord bot to post a challenge announcement to the hub's channel. Bot monitors reactions, completions, and interactions, then emits back to Aura to update challenge state. The bot becomes a bidirectional event bridge.

Socket.io gateway — rooms by hub ID
Every connected client joins a room keyed to their hub: hub:{hubId}. The bot's event consumer and the relay service emit to these rooms. All realtime state (messages, presence, challenge events) flows through this single bus.
// Client joins hub room on connect socket.on('hub:join', async ({ hubId, token }) => { const player = await verifyJWT(token); await verifyHubAccess(player.id, hubId); socket.join(`hub:${hubId}`); });
Redis Streams — ordered, replayable message backbone
Replace the direct webhook→socket pipe with Redis Streams. Two consumer groups read from the same stream: socket-broadcaster (pushes to Socket.io rooms) and discord-relay (posts to Discord webhooks). New joiners can replay recent history from the stream without hitting Discord's API.
// Producer: inbound message from Aura client await redis.xadd(`hub:${hubId}:messages`, '*', { authorId, content, tempId, timestamp: Date.now() }); // Consumer group: broadcast to room const msgs = await redis.xreadgroup('socket-broadcaster', 'node-1', { key: `hub:${hubId}:messages`, id: '>' }); msgs.forEach(m => io.to(`hub:${hubId}`).emit('message', m));
Message relay — bidirectional bridge
Outbound: consumes from Redis Stream, posts to Discord webhook. Inbound: Discord bot emits gateway events (new messages from other clients) → normalized → pushed to Redis Stream → broadcast to Aura room. Signature verification on all webhook payloads.
// Outbound: Aura → Discord async function relayToDiscord(hubId, message) { const { webhookUrl } = await HubDiscordMapping.findOne({ auraHubId: hubId }); await fetch(webhookUrl, { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ username: message.authorName, content: message.content, }), }); }
Normalization layer — Discord format → Aura-native
Bot receives raw Discord message objects. Normalize them before they touch Aura: resolve mentions to Aura player IDs, convert Discord CDN URLs to Aura media proxies, transform embeds to Aura card components, map Discord sticker IDs to Aura media assets.
function normalizeDiscordMessage(msg, mapping) { return { id: msg.id, content: resolveMentions(msg.content, mapping), author: resolveAuthor(msg.author.id, mapping), attachments: msg.attachments.map(proxyCDNUrl), sticker: msg.sticker_items?.[0] ? resolveStickerToAura(msg.sticker_items[0].id) : null, timestamp: msg.timestamp, }; }
LiveKit SFU — self-hosted WebRTC media server
Deploy LiveKit on your own infra (Docker image, ~200MB). Your backend issues short-lived room tokens keyed to hub:{hubId}:voice. LiveKit handles STUN/TURN, NAT traversal, encrypted audio streams, adaptive bitrate — everything Discord's client runtime does, but controlled by you.
// Issue LiveKit token for hub voice import { AccessToken } from 'livekit-server-sdk'; app.get('/api/hubs/:id/voice-token', auth, async (req, res) => { const at = new AccessToken(LK_API_KEY, LK_API_SECRET, { identity: req.player.id, ttl: '1h', }); at.addGrant({ roomJoin: true, room: `hub:${req.params.id}` }); res.json({ token: await at.toJwt() }); });
Presence service — Discord voice states in Aura UI
Bot gateway VOICE_STATE_UPDATE events flow into this service, which writes to a Redis hash hub:{hubId}:voice-members and emits to Socket.io rooms. Aura renders the live member list with mute/deafen states without any Discord UI. Completely native Aura presence.
// Redis hash: who's in voice right now await redis.hset(`hub:${hubId}:voice-members`, playerId, JSON.stringify({ displayName, avatar, muted, deafened, speaking: false, })); // Speaking indicator via Discord's SPEAKING gateway event client.ws.on('SPEAKING', ({ user_id, speaking }) => { io.to(`hub:${hubId}`).emit('presence:speaking', { userId: user_id, speaking }); });
Hub provisioner — automated Discord resource setup
When a new Aura hub is created, this service calls the Discord bot to: create a guild category, create a text channel, create a voice channel, register a webhook on the text channel. On success, writes the HubDiscordMapping record. Status transitions: PROVISIONING → ACTIVE.
async function provisionHubInDiscord(hub) { const category = await guild.channels.create({ name: hub.name, type: ChannelType.GuildCategory }); const text = await guild.channels.create({ name: 'general', type: ChannelType.GuildText, parent: category.id }); const voice = await guild.channels.create({ name: 'voice', type: ChannelType.GuildVoice, parent: category.id }); const webhook = await text.createWebhook({ name: 'Aura Relay' }); await HubDiscordMapping.create({ auraHubId: hub._id, discordGuildId: guild.id, discordCategoryId: category.id, discordChannelId: text.id, discordVoiceChannelId: voice.id, webhookUrl: webhook.url, syncStatus: 'ACTIVE', }); }
JWT middleware — identity across all surfaces
Aura JWT is verified on every socket event and HTTP request. The bot maintains a discordId ↔ auraPlayerId mapping in HubAccessState. When a Discord event arrives, the normalizer resolves the Discord user ID to an Aura player — so messages always carry Aura identity, not Discord identity.

Bot gateway — all inbound events flow here
Your Discord.js bot maintains a persistent WebSocket to Discord's gateway. It listens for: MESSAGE_CREATE (relay to Aura), VOICE_STATE_UPDATE (presence mirror), GUILD_MEMBER_ADD/REMOVE (access sync), and custom interaction events (challenge completions). This is the only persistent connection to Discord.
Webhooks — Aura messages written to Discord
Per-channel webhooks are created by the Hub Provisioner during hub setup. The message relay service posts to these URLs when an Aura user sends a message. Discord becomes a durable ledger of all hub communication, accessible from native Discord clients if needed.
Guild API — provisioning and access control
Used only by the Hub Provisioner and access sync services. Creates categories/channels on hub creation, manages permission overwrites for private hubs, handles member role assignment. Not involved in the message hot path — keeps the relay latency clean.
Discord voice — graceful fallback only
With LiveKit SFU handling browser voice, Discord's voice channels are demoted to fallback. If LiveKit is unavailable, Aura shows a "join via Discord" button that deep-links with the discord:// protocol handler as a last resort. Most users will never see this path.
